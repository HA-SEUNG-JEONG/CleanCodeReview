### 😀 책에서 기억하고 싶은 내용을 써보세요.
- 클래스나 객체 이름은 명사,명사구 메소드 이름은 동사,동사구가 적합하다.
- **기발한 이름은 피하라.** 저자와 유머 감각이 비슷한 사람만, 그리고 농담을 기억하는 동안만 이름을 기억한다. 재미난 이름보다 의도가 분명하게 드러나는 이름을 선택해야 한다.
- **한 개념에 한 단어를 사용하라.** 똑같은 메소드를 클래스마다 fetch,retrieve,get으로 각각 부르게 되면 혼란스럽다. 어느 클래스에서 어느 이름을 썼는지 헷갈리기 때문이다.
- **말장난을 하지마라.** 같은 맥락이 아닌데도 '일관성'을 고려하여 특정 단어를 다시 사용한다면 그것은 말장난이다. 의미를 해독할 책임이 독자에게 있는 논문 모델이 아니라 저자에게 있는 잡지 모델이 바람직하다.
- **의미 있는 맥락을 추가하라.** 스스로 의미가 분명한 이름이 없지 않다. 하지만 대다수는 그렇지 못하기 때문에 클래스,함수, 네임스페이스에 넣어 맥락을 부여한다. 모든 방법이 실패하면 마지막 수단으로 **접두어**를 추가한다.
- **불필요한 맥락을 없애라.** 고급 휘발유 충전소(Gas Station Deluxe)라는 애플리케이션을 짠다고 가정하면, 모든 클래스 이름을 GSD로 하는 것은 전혀 바람직하지 못하다. IDE에서 G를 입력하면 G로 시작하는 모든 클래스를 열거하기 때문이다.

### 😀 책에서 기억하고 싶은 내용을 써보세요.
- **작게 만들어라.** : if/else/while 문 등에 들어가는 블록은 한 줄이어야 한다. 대개 거기서 함수를 호출하며, 블록 안에서 함수 이름을 적절하게 짓는다면 코드도 이해하기 쉬워진다.
- 함수는 한 가지만 해야하고 그 한 가지를 잘해야 한다.
- 코드는 위에서 아래로 이야기처럼 읽혀야 좋다. 한 함수 다음에는 추상화 수준이 한 단계 낮은 함수가 온다.
- **서술적인 이름을 사용하라** : 좋은 이름이 주는 가치는 아무리 강조해도 지나치지 않다. 길고 서술적인 이름이 짧고 어려운 이름보다 좋다. 또한 서술적인 이름을 사용하면 개발자 머릿속에서도 설계가 뚜렷해지므로 코드를 개선하기 쉬워진다.
- 함수에서 이상적인 인수 개수 0개이며, 3개는 피하는 편이 좋다. 인수가 들어가면 개념을 이해하기가 어려워진다. 코드를 읽는 사람에게는 `includeSetUpPageInto(new PageContent)` 보다 `includeSetUpPage()`가 이해하기 더 쉽다.
- **부수 효과를 일으키지 마라.** : 함수를 한 가지만 잘해야 하는데 자칫 잘못 호출하게 되면 의도치 않은 일이 발생하게 된다.

</br>


### 🤔 오늘 읽은 소감은? 떠오르는 생각을 가볍게 적어보세요.
프로젝트를 수행할 때 함수에 인수를 넘겨서 코드를 진행시킨 적이 있었는데 나중에 코드를 다시 보면 한 번에 이해가 바로 안된 적이 몇 번 있었다. 인수가 없는 경우에는 확실히 이해하기가 쉬웠다.
코드를 돌아가게끔 하고 나중에 리팩토링을 할 때가 제일 두렵고 어려운 것 같다. 분해하기 시작하면 다 망가지지 않을까라는 생각 때문인 것 같다.
1장에서 나온 내용처럼 부딪혀보고 연습하고 반복하는 수 밖에 없을 것 같다.

### 😀 책에서 기억하고 싶은 내용을 써보세요.
- **명령과 조회를 분리하라.** 함수는 뭔가를 수행하거나 뭔가에 답하거나 둘 중 하나만 해야한다. "set"이라는 단어가 동사인지 형용사인지 코드만 봐서는 분간하기 어렵다. 따라서 명령과 조회를 분리해서 애초부터 혼란을 뿌리뽑아야 한다.
- **오류 코드보다 예외를 사용하라.** 명령 함수에서 오류 코드를 반환하는 방식은 명령/조회 분리 규칙을 미묘하게 위반한다. 자칫하면 if 문에서 명령을 표현식으로 사용하기 쉬운 탓이다.
- try/catch 블록도 원래 추하다. 코드 구조에 혼란을 야기하고, 정상 동작과 오류 처리 동작을 뒤섞는다. 따라서 try/catch 블록을 별도의 함수로 분리하는 편이 낫다.
- **오류 처리도 한 가지 작업이다.** 함수는 한 가지 작업만 해야한다. 오류 처리도 하나의 작업에 속한다. 그렇기 때문에 오류를 처리하는 함수는 오류만 처리해야 마땅하다.
- 이전에 설명한 규칙을 잘 따른다면 길이가 짧고, 이름이 좋고 체계가 잘 잡힌 함수가 나올 것이다. 하지만 진짜 목표는 시스템이라는 이야기를 풀어가는 데에 있다. 작성하는 함수가 분명하고 정확한 언어로 깔끔하게 맞아 떨어져야 이야기를 풀어가기가 쉬워진다.

const merry = document.querySelector(".js-clock");

function getClock() {
const christmas = new Date("2021, 12, 25");
const date = new Date();
const timeGap = christmas - date;

const xDay = Math.floor(timeGap / (1000 * 60 * 60 * 24));
const xHours = Math.floor(
(timeGap - xDay * 1000 * 60 * 60 * 24) / (1000 * 60 * 60)
);
const xMinutes = Math.floor((timeGap % (60 * 60 * 1000)) / (60 * 1000));
const xSeconds = Math.floor((timeGap % (60 * 1000)) / 1000);

const day = String(xDay).padStart(2, "0");
const hours = String(xHours).padStart(2, "0");
const minutes = String(xMinutes).padStart(2, "0");
const seconds = String(xSeconds).padStart(2, "0");

merry.innerText = `${day}d ${hours}h ${minutes}m ${seconds}s`;
}

getClock();
setInterval(getClock, 1000);

  return String(arg);

  const christmas = new Date("2022, 12, 25");
  const date = new Date();
  return christmas-date;
}

  const DisplayTime=getClock();
  merry.innerText=DisplayTime;
}


function getClock() {
  const christmasremainingtime=ChristmasTimeGap();
  const christmasremainingDay = Math.floor(christmasremainingtime / DaysToMillisec);
  const christmasremainingHours = Math.floor((christmasremainingtime - christmasremainingDay * DaysToMillisec) / HoursToMillisec);
  const christmasremainingMinutes = Math.floor((christmasremainingtime % HoursToMillisec) / MinutesToMilisec);
  const christmasremainingSeconds = Math.floor((christmasremainingtime % MinutesToMilisec) / 1000);

  return `${RemainToChristmas(christmasremainingDay)}days ${RemainToChristmas(christmasremainingHours)}hours ${RemainToChristmas(christmasremainingMinutes)}minutes  ${RemainToChristmas(christmasremainingSeconds)} seconds`;

DisplayRemainTime();

- **null을 전달하지 마라.** 애초에 null을 넘기지 못하도록 금지하는 정책이 합리적이다. 만약 인수로 null이 넘어오면 그 코드는 문제가 있다는 말이다.
- 깨끗한 코드는 읽기도 좋아야 하지만 **안정성**도 높아야 한다. 오류 처리를 프로그램 논리와 분리하면 독립적인 추론이 가능해지며 유지보수도 쉬워진다.

- 자바 첫 버전에서는 확인된 예외가 멋진 아이디어로 여겨졌지만 지금은 안정적인 소프트웨어를 제작하는 요소로 확인된 예외가 반드시 필요하지는 않다는 사실이 분명해졌다.
